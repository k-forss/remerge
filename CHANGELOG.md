## [0.0.0] ‚Äî 2026-03-20

Initial release ‚Äî distributed binary package builder for Gentoo Linux.

### Added

#### Core

- CLI (`remerge`) ‚Äî drop-in `emerge` wrapper that submits workorders to a
  remote build server, streams progress over WebSocket, and installs the
  resulting binary packages locally
- Server (`remerge-server`) ‚Äî HTTP/WS API for workorder lifecycle,
  Docker-based worker orchestration, and binpkg hosting
- Worker (`remerge-worker`) ‚Äî containerised build agent that applies
  forwarded portage configuration and runs `emerge`
- Shared types crate (`remerge-types`) for API, portage, and workorder types

#### Build pipeline

- Structured emerge output parsing ‚Äî `REMERGE_EVENT:` JSON lines for
  per-package start/finish/failure events with timing
- Failure pattern detection for missing dependencies, USE flag conflicts,
  and fetch failures
- Post-build binpkg scan with SHA-256/size verification and automatic
  `Packages` index regeneration
- Worker concurrency via tokio semaphore with FIFO queuing (`max_workers`)
- Configurable `parallel_jobs` / `load_average` with CPU auto-detection
- Worker binary injection into Docker images (`worker_binary` config)
- Idle image reaper ‚Äî removes unused Docker images after
  `worker_idle_timeout`, preserving MRU per `(CHOST, profile, GCC)` tuple

#### Portage integration

- Configuration snapshotting: `make.conf`, `package.use`,
  `package.accept_keywords`, `package.license`, profile, world
- `-march=native` translation for cross-compilation compatibility
- Crossdev toolchain detection with automatic `emerge-<CHOST>` selection
- `@world` / `@system` set expansion to individual atoms
- Atom validation with shell metacharacter rejection
- `cpuid2cpuflags` integration (`CPU_FLAGS_*` forwarding)
- `USE_EXPAND` forwarding (`VIDEO_CARDS`, `PYTHON_TARGETS`, etc.)
- Extra `make.conf` variable passthrough with input sanitisation
- OpenPGP signing of `.gpkg` packages via portage's `binpkg-signing` feature

#### State & lifecycle

- Workorder and client registry persisted to JSON in `state_dir`; incomplete
  workorders marked failed on restart
- TTL eviction after configurable `retention_hours` (default 24 h)
- Configurable `max_retained_workorders` cap (default 1 000), oldest-first
- Cancel running builds ‚Äî stops the Docker container and broadcasts status
  over WebSocket
- Old-version binpkg cleanup ‚Äî keeps the N most recent versions per package

#### Authentication & identity

- Three auth modes: `none`, `mtls`, `mixed`
- mTLS via reverse-proxy forwarded certificate fingerprint headers
- Certificate registry with flexible fingerprint normalisation
- Client roles: `main` (push config) / `follower` (build only)
- Configurable client ID: `--client-id`, `REMERGE_CLIENT_ID`, or
  `/etc/remerge.conf`
- One active workorder per client enforced server-side
- Read endpoints scoped to the authenticated client in `mtls`/`mixed` modes

#### Observability

- Prometheus metrics at `GET /metrics` ‚Äî workorder counters, active builds,
  build duration histogram, queue depth, binpkg disk usage
- Background disk-usage monitor with configurable warning threshold
- JSON structured logging via `log_json` / `REMERGE_LOG_JSON`
- Health endpoint (`GET /api/v1/health`)
- Admin endpoints (`GET /api/v1/clients[/{id}]`)

#### API

- `POST /api/v1/workorders` ‚Äî submit workorder
- `GET  /api/v1/workorders` ‚Äî list workorders
- `GET  /api/v1/workorders/{id}` ‚Äî status and result
- `DELETE /api/v1/workorders/{id}` ‚Äî cancel pending or running build
- `WS  /api/v1/workorders/{id}/progress` ‚Äî real-time build log stream
- `GET  /api/v1/info` ‚Äî server info, auth mode, signing status
- `GET  /binpkgs/...` ‚Äî portage binhost repository
- WebSocket scheme detection via `X-Forwarded-Proto`

#### Networking

- Optional native TLS via `[tls]` config (`tokio-rustls`, no reverse proxy
  required)
- `rustls` backend ‚Äî no OpenSSL build dependency
- CLI `--force` flag to skip local VDB check and force rebuild

#### Deployment

- Multi-stage Docker image for the server
- docker-compose example with state-persistence volume
- Gentoo overlay with five packages: `remerge`, `remerge-bin`,
  `remerge-server`, `remerge-server-bin`, `openpgp-keys-remerge`
- OpenRC init/conf and hardened systemd unit for `remerge-server`
- Live `9999` ebuilds checked in; versioned ebuilds generated by the RC
  workflow

#### CI/CD & supply chain

- GitHub Actions CI ‚Äî fmt, clippy, test, security audit
- Three-phase release signing: multi-arch build + SLSA attestation ‚Üí
  VirusTotal scan ‚Üí PGP detached signatures + clear-signed release body
- RC workflow: push `rc-X.Y.Z` ‚Üí changelog + versioned ebuilds ‚Üí merge ‚Üí
  auto-tag ‚Üí release
- Multi-arch binaries ‚Äî amd64, arm64, armv7, ppc64, riscv64 (CLI);
  amd64, arm64 (server)
- `cargo deny` for licence and advisory auditing
- Dependabot for Cargo, Actions, and Docker dependencies
- Branch/tag rulesets: signed commits, linear history, protected refs

### Known limitations

- No integration or end-to-end tests ‚Äî unit tests only
- OpenTelemetry trace propagation not implemented
- `remerge_queue_depth` metric defined but not wired to the queue processor
- `@system` set expansion does not resolve virtual packages
